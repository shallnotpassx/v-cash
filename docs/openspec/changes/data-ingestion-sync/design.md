# Design

## 目标

把“拉数据”做成明确的后台能力，而不是隐藏在查询接口背后的副作用。

## 核心决策

### 1. 触发方式

一期以“按指定代码手动拉取”为主。

定时同步只作为后续增强入口，不抢主流程地位。

### 2. 数据源模式

外部数据源按“策略 + 兜底”方式组织：

- 先尝试首选免费源
- 失败后按预设顺序切换下一源
- 返回首个成功结果

### 3. 同步记录

每次同步都要留下记录，至少包含：

- 同步对象
- 触发方式
- 使用的数据源
- 开始时间 / 结束时间
- 成功或失败状态
- 失败摘要

## 依赖边界

这个能力域依赖 `stock-registry`，因为同步必须知道本地股票身份。

它不直接拥有财务筛选和公告展示规则，只负责把原始结果写到对应能力域需要的存储位置。

## 风险与约束

- 免费源稳定性不可完全控制，需要把失败信息显式暴露出来
- 不同市场和不同源的字段名称可能不一致，需要在适配器层标准化
