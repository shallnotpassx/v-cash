# 公告中心规范

## 新增需求

### 需求：维护独立的公告元数据

系统必须将公告元数据与财务记录分开维护。

#### 场景：保存一条公告元数据

- **给定** 系统发现了一条属于某只股票的公告
- **当** 保存该公告记录时
- **那么** 系统保存股票引用、公告标题、公告类型、披露日期、来源站点和原文链接

### 需求：提供本地公告列表、筛选与详情查询

系统必须基于本地库存提供公告列表、筛选和详情查询能力。

#### 场景：按条件筛选本地公告

- **给定** 本地存储中已经存在公告元数据
- **当** 用户按股票、日期范围或公告类型筛选时
- **那么** 系统从本地存储返回匹配的公告列表

#### 场景：查看单条公告详情

- **给定** 本地存储中已经存在目标公告记录
- **当** 用户查看该公告的详情时
- **那么** 系统返回该公告的元数据和可访问的原文信息

### 需求：默认以外部链接保留公告原文

系统必须默认通过外部链接保留公告原文，而不是强制缓存文档文件。

#### 场景：打开公告原文

- **给定** 某条公告已经保存了原文链接
- **当** 用户打开该公告的原文时
- **那么** 系统使用已保存的外部链接访问原始文档
- **并且** 默认不要求将 PDF 原文缓存到本地

### 需求：为人工解析保留文档入口

系统必须为后续人工查看或 PDF 解析保留稳定的文档访问入口。

#### 场景：从公告进入人工解析链路

- **给定** 结构化财务数据缺失，用户需要查看公告或年报原文
- **当** 用户从公告记录进入文档入口时
- **那么** 系统提供可用于人工查看或后续解析的文档访问入口
